深入理解Kafka优化的核心参数

¶性能测试

¶权衡throughput VS latency 、durability VS availability

  • 优化throughput

:start: broker端

  1. 确定topic的分区个数

    topic分区是并行处理的最小单元。消息的并行发送和接收都是基于分区的。通常而言，topic的分区越多，并行处理能力越强，吞吐量越大。我们需要根据producer和consumer的吞吐量去确定分区的个数。

    这篇文章详细讲解了如何确定分区个数

生产者端

  1. 批量策略
    生产者通过将多个消息以一个消息的批量方式发送消息。两个核心参数
    batch.size和linger.ms；但是会影响延时，因为必须消息不是立即发送的，而是需要收集batch.size个消息或者收集消息的等待时间超过linger.ms。设置compression.type（支持lz4, snappy, and gzip），批量消息将通过压缩的方式压缩成一个消息，所以batch.size越大，压缩率越高，吞吐量越大。

  2. acks方式
    acks=1 leader broker收到消息后就返回ack，而不用等所有的followers都收到消息。

  3. 发送时机
    buffer.memory
    max.block.ms
    二者满其一将发送

消费者端

  1. fetch.min.bytes
    这个值越大，从broker获取消息的次数越少，会减轻broker的CPU压力；但是会影响延时时间。达到fetch.min.bytes或者超过fetch.max.wait.ms时，才会消费消息。

GC
gc会导致应用暂停，所以长时间的GC时间会严重影响吞吐量；严重的情况下会导致broker失败，比如长时间的GC会导致ZooKeeper session超时，ZooKeeper会认为该broker不可用。

GC该如何优化参考GC调优

小结

Producer: • batch.size: increase to 100000 - 200000 (default 16384) • linger.ms: increase to 10 - 100 (default 0) • compression.type=lz4 (default none) • acks=1 (default 1) • retries=0 (default 0) • buffer.memory: increase if there are a lot of partitions (default 33554432)
Consumer: • fetch.min.bytes: increase to ~100000 (default 1)

  • 优化latency

消息复制会影响延时，复制消息的线程个数（num.replica.fetchers）会影响延时，num.replica.fetchers=1（默认值），当followers无法跟上leader时，可以将该值调大。

linger.ms=0 即时发送
compression.type=none 压缩会使用CPU、降低网络带宽使用率
acks=0 异步发送，发送者无需等待broker的ack
etch.min.bytes=1 有消息就接收

小结

Producer: • linger.ms=0 (default 0) • compression.type=none (default none) • acks=1 (default 1) Broker: • num.replica.fetchers: increase if followers can’t keep up with the leader (default 1) Consumer: • fetch.min.bytes=1 (default 1)

  • 优化durability
    replication.factor=3 可以保证在2个broker不可用的情况下，消息不丢失。

acks=-1
retries 自动重试

重试的副作用

  1. 重复消息；集群出现短暂失败时可能会导致重复消息，需要消费者端自己处理重复消息；另外exactly once semantics (EOS) 的功能还在开发中，这种机制可以保证无重复消息。
  2. 导致消息重排；为了保证消息有序并且允许重新发生失败的消息，需要将max.in.flight.requests.per.connection设置为1，从而保证任何时候只有一个消息发送到broker。

unclean.leader.election.enable=false 不允许不在ISR列表中的broker参加leader的选举，否则会导致已经提交但是还未复制的消息的丢失。

log.flush.interval.ms和log.flush.interval.messages
消息写入broker是直接写入OS的PageCache的，然后由OS周期性地将数据从PageCache刷到磁盘；为了保证不丢消息，就需要将消息同步刷到磁盘，所以可以针对这类topic设置log.flush.interval.messages=1

消费者是通过更新offsets来表明offsets之前的消息已经被消费了；当消费者的poll()执行完时，会自动commit offsets。但是如果poll作为业务事务中的一部分的时候，为了保证可靠性，必须在事务提交之后才能提交offsets，所以需要将auto.commit.enable=false，并且显示地调用
commitSync() or commitAsync()

小结

Producer: • replication.factor: 3, configure per topic • acks=all (default 1) • retries: 1 or more (default 0) • max.in.flight.requests.per.connection=1 (default 5), to prevent out of order messages Broker: • default.replication.factor=3 (default 1) • auto.create.topics.enable=false (default true) • min.insync.replicas=2 (default 1); topic override available • unclean.leader.election.enable=false (default true); topic override available • broker.rack: rack of the broker (default null) • log.flush.interval.messages, log.flush.interval.ms: for topics with very low throughput, set message interval or time interval low as needed (default allows the OS to control flushing); topic override available Consumer: • auto.commit.enable=false (default true)

  • 优化availability

Broker: • unclean.leader.election.enable=true (default true); topic override available • min.insync.replicas=1 (default 1); topic override available • num.recovery.threads.per.data.dir: number of directories in log.dirs broker启动的时候(default 1)
Consumer: • session.timeout.ms: as low as feasible (default 10000) broker检测consumer是否可用的时间，消费者发生异常时broker会检测到失败并且重新rebalance。消费者失败要么是硬件失败导致，要么是软件失败导致。软件失败最场景的两个场景：
poll()处理时间过长；或者是GC暂停时间太长

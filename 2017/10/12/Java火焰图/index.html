Java混合模型火焰图

火焰图(Flame Graph)

火焰图是一个用于分析包含stack traces的profile数据的可视化工具，通过它可以快速准确地定位到热点code-paths，是分析CPU瓶颈的“懒人”工具。
火焰图有不同的类型：CPU、Memory、Off-CPU、Hot/Cold以及Differential

下图就是一个CPU的火焰图

理解火焰图

对CPU火焰图的解读

  • 小方块
    每个小方块代表一个方法（就是一个stack frame），每小方块的宽度表示该方法在采样期间出现的频率，宽度越宽表示该方法出现的频率越高
  • Y轴
    表示调用堆栈，体现出了stack深度，每个顶端的小方块代表正在使用CPU（on-CPU）的方法
  • X轴
    标示采样总量，会对调用栈进行合并，然后根据方法的字母顺序进行排列，X轴并不代表使用CPU的时间长短，而只是说明该方法出现的频率高低，宽度越宽表示该方法出现的频率越高。国内不少的文章中都将X轴看做是使用CPU的时间长短，这是在误导读者。
  • 颜色
    小方块的颜色并没有特定的含义，是随机选取的暖色色调

如何生成火焰图

任何包含stack traces的profile数据都可以生成火焰图，以下profiling工具可以生成的profile数据

Linux: perf_events(简称perf) , eBPF, SystemTap, and ktap
Solaris, illumos, FreeBSD: DTrace
Mac OS X: DTrace and Instruments
Windows: Xperf.exe

生成火焰图只需要两步：采集stack traces数据、使用FlameGraph的脚本生成火焰图

通过perf的命令和脚本，就可以直接生成火焰图。下图显示了perf的工作流程，其中perf record展示了生成火焰图的主要步骤

通过perf生成perf.data文件后，就可以通过FlameGraph（Stack trace可视化工具）的脚本生成火焰图

perf script: 读取perf.data的文件内容，并以多行的形式生成stacks

flamegraph.pl: 根据折叠之后的stacks信息生成svg格式的火焰图

该svg格式的火焰图内嵌了JS代码，通过浏览器打开，自带了交互功能：鼠标移动显示方法信息、可以点击放大或者缩小调用栈以及通过表达式查找对应的方法名。

火焰图的缺陷

System profilers的火焰图不显示Java的调用栈（stack）和方法(method)，所以无法定位Java的哪个方法比较耗CPU。

为了解决这两个问题，Brendan Gregg对火焰图工具进行了改进，创造出一种Java混合模型火焰图

什么原因导致无法显示method以及完整的stack呢？

  • 为什么不显示method

原因在于JIT在编译method时，并没有暴露出一个symbol table，导致profiler工具无法获取具体的method

  • 为什么statck不完整

原因在于在x86 (RBP on x86–64)上，JVM使用frame指针寄存器作为通用的寄存器，这种寄存器打破了传统的stack的运行方式，导致stack不完整

如何解决显示method名称的问题

要想解决显示method名称这个问题，首先要能够收集到symbols信息，然后再从这些信息中识别出method名称

  • 如何收集symbols信息

2014年Johannes Rudolph写了一个名为perf-map-agent的开源Java代理，通过这个代理可以对Java应用生成/tmp/perf-PID.map文件。这个文件里面记录了所有symbol的地址（十六进制）、大小和名称。所以通过perf-map-agent可以收集到symbols信息。

perf-map-agent是一个开源的Java代理，它包括一个用C写的Java代理以及一个可以将这个代理attach到Java进程的简易Java代码

  • 如何从symbols中识别method

2009年，perf已经支持了JIT symbol，通过这个特征perf会默认地查找/tmp/perf-$PID.map文件，从该文件中解析出function symbol的信息，从而识别出method名称。

如果找不到function symbol信息，当前版本的perf脚本的处理方式是采用library的名字或者perf-$PID.map文件名作为method的名字

所以结合perf以及perf-map-agent生就可以在火焰图上显示出Java method。

如何解决stack不完整的问题

多年前为了优化编译器，gcc编译器使用了frame pointer，这种技术打断了stack的调用栈，使得stack不完整。但是如果在使用gcc时，使用了-fno-omit-frame-pointer选项，那么就会显示完整的stack。然而JVM并没有对应的选项来控制是否显示完整的stack。

这个特征能不能在JDK中实现呢？因此如果JVM能够实现通过一个选型控制是否显示完整的stack，那么这个问题就解决。

技术大咔Brendan出于好奇，他在OpenJDK8上开发了一个可以工作的原型，他期望能够在Oracle JDK中实现这个特征，于是他在hotspot-compiler-dev开发邮件组中发了一封邮件，表达了自己不是资深的hotspot工程师，希望有人可以优化或者重写这个原型。

于是乎这个特性分别记录在JDK9的JDK-8068945和JDK8的JDK-8072465中。Oracle的工程师Zoltán Majó为了重写了这个patch做了大量的工作，四个月后终于完成，将这个特性集成在JDK9和JDK8（JDK8 update 60 build 19）的发布版本里，使用选项 -XX:+PreserveFramePointer控制是否显示完整的stack。

因此通过打开 -XX:+PreserveFramePointer就解决了stack不完整的问题，前提是JDK的版本必须大于或者等于Java 8 update 60 build 19

  • stack是否完整的火焰图对比


火焰图对系统性能的损耗

-XX:+PreserveFramePointer对性能的影响
Orcale的性能测试团队测试的结果是性能下降2%-5%（SPECjvm2008-Derby* -2% to -5%），出于对性能的考虑，他们建议默认配置不启用PreserveFramePointer
Netflex的性能测试结果是性能下降0-3%
